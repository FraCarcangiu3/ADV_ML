================================================================================

INTERPRETAZIONE DEI LOG
--------------------------------------------------------------------------------

1. Log di Bootstrap (sempre presente):
   [AUDIO_OBF] enabled=<0|1> [from=<ENV|CLI>] [use_pitch=<0|1>] ...
   
   - enabled=0: Sistema disabilitato, nessun processing
   - enabled=1: Sistema abilitato, processing attivo
   - from=ENV: Configurazione da variabile d'ambiente AC_AUDIO_OBF
   - from=CLI: Configurazione da argomento --audio-obf
   - use_pitch/noise/tone: Flag per trasformazioni (Step 1: tutte a 0)

2. Log per Singolo Suono (solo se enabled=1):
   [AUDIO_OBF] <soundname> → pitch:±N cents, noise:SNR=X.X dB, tone:FFFF Hz @ LL.L dB
   
   - <soundname>: Nome del file audio (es. "player/footsteps.ogg")
   - pitch:±N cents: Shift di pitch in cents (Step 1: sempre 0)
   - noise:SNR=X.X dB: SNR del rumore aggiunto (Step 1: sempre 0.0)
   - tone:FFFF Hz @ LL.L dB: Tono sinusoidale aggiunto (Step 1: sempre 0 Hz @ 0.0 dB)

3. Formato Parsabile:
   I log sono progettati per essere facilmente parsati da script:
   
   grep "^\[AUDIO_OBF\]" logfile.txt
   
   Per statistiche:
   grep -c "AUDIO_OBF.*→" logfile.txt  # conta suoni processati


================================================================================

NOTE STEP 1
--------------------------------------------------------------------------------

In Step 1, il framework è SOLO infrastruttura + logging.

NESSUNA MODIFICA al buffer PCM viene applicata, quindi:
- L'audio suona identico a prima
- I log mostrano pitch:+0, noise:0.0, tone:0 Hz
- Performance impact: minimo (solo una chiamata di funzione no-op)

Gli step successivi implementeranno le trasformazioni reali:
- Step 2: Pitch shifting effettivo (use_pitch=1, pitch_cents!=0)
- Step 3: Noise injection (use_noise=1, noise_snr_db>0)
- Step 4: Tone injection (use_tone=1, tone_freq_hz>0)
